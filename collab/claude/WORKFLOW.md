# Claude Sidecar Workflow (Writing + Design Critique Lane)

This folder is the lightweight collaboration protocol between:
- **Codex** (implementation/orchestration)
- **Claude** (writing/design critique/planning sidecar)

## Goal

Use Claude to improve:
- writing quality
- stakeholder brief tone/structure
- page layout/readability critique
- planning alternatives

without adding Claude directly into the production agent pipeline.

## Inputs for Claude

Start with:
- `CLAUDE.md`
- `data/abivax/claude_lane_queue.json`
- `collab/claude/prompts/NEXT_PROMPT.md` (preferred, generated by Codex)
- `collab/shared/CURRENT_OPERATING_BRIEF.md` when present
- `collab/shared/SOURCE_SYSTEM_FLOW.md` when present
- `collab/shared/CONTEXT_WINDOW_PROTOCOL.md` when present
- `data/abivax/current_context.json` when present

Optional supporting files (only as needed):
- specific page/component files referenced by a queue item
- targeted JSON artifacts referenced by a queue item

## Required Output Format (save to repo)

Write each response to:
- `collab/claude/outputs/YYYY-MM-DD_<topic>.md`

Recommended structure:
1. `Queue item(s) reviewed`
2. `What is wrong (specific)`
3. `Proposed rewrite/structure`
4. `Why this is better`
5. `What Codex should implement` (concrete and minimal)

## Boundaries

Claude should not:
- directly modify canonical program JSONs unless explicitly instructed by Mike
- change agent scripts/waterfall logic
- add front-end workflow mechanics (buttons/forms) for orchestration

Claude can:
- propose wording/layout/structure
- draft text blocks
- critique page readability
- identify signal/noise problems
- implement front-end UI/layout code when Mike explicitly asks for direct coding in scope files

## Direct UI Implementation Lane (Claude -> Codex Gate)

Use this lane when Mike wants Claude to code page/UI improvements directly.

1. Claude implements only the requested UI/layout changes in scoped front-end files.
2. Claude reports completion with:
   - changed files list
   - short summary of UI changes
   - explicit line: `READY_FOR_CODEX_REVIEW`
3. Mike tells Codex to review and verify.
4. Codex performs:
   - diff/code review
   - `npm run build`
   - `npm run verify:spine-smoke`
   - stabilization patches as needed
5. Codex returns final accepted state.

### Heuristic (When to Use This Lane First)

Use Claude-first UI implementation by default when the request is primarily:
- visual hierarchy/readability
- layout restructuring
- reducing cognitive load
- ownership clarity between human actions vs system queue actions

Use Codex-first implementation by default when the request is primarily:
- data model/schema changes
- pipeline/agent orchestration
- ingestion/enrichment/detection logic
- backend correctness or cross-surface regressions

## Handoff Back to Codex

Mike can tell Codex:
- `review Claude output collab/claude/outputs/<file>.md`
- `review Claude UI implementation` (after Claude posts `READY_FOR_CODEX_REVIEW` with changed files)

## Smoothest Loop (Recommended)

1. Codex runs:
   - `npm.cmd run agent:claude-lane-queue`
   - `npm.cmd run agent:claude-next-prompt`
2. Mike tells Claude:
   - `Open collab/claude/prompts/NEXT_PROMPT.md and execute it.`
   - If present, also read `collab/shared/CURRENT_OPERATING_BRIEF.md` first.
   - If present, use `data/abivax/current_context.json` as the declared context window for the session.
3. Claude writes output to `collab/claude/outputs/...`
4. Mike tells Codex:
   - `review Claude output <path>`

Codex will then:
- decide what to adopt
- implement changes
- rerun relevant waterfall stages/checks

## Single Active Window Rule

Treat Claude and Codex as separate context windows sharing one repo memory layer.

- Once Mike switches from Codex to Claude, Claude should assume it is the active window.
- Claude should not rely on Mike bouncing back to Codex casually for missing context.
- If Claude needs new Outlook/Desktop/source-system information, Claude should finish the current unit of work and hand off back to Codex through a deliberate re-entry note.
- See `collab/shared/CONTEXT_WINDOW_PROTOCOL.md`.

## Smoothest Direct UI Loop (Alternative)

1. Mike tells Claude to implement the queued UI/layout changes directly.
2. Claude finishes and posts:
   - changed files
   - `READY_FOR_CODEX_REVIEW`
3. Mike tells Codex:
   - `review Claude UI implementation` (optionally paste changed files list)
4. Codex reviews, verifies, patches, and finalizes.
